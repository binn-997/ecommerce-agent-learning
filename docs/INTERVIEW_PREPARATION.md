## 3. 三条核心链路

### 3.0 蔚蓝思自动化工具开发经历怎么讲

这段工作经历是“真实业务应用”，个人项目是“把经验工程化”。面试时先讲工作中的实际流程，再说明哪些问题促使你开发 Amazon AI Platform，不要把两者混成同一个已经上线的平台。

推荐用下面的链路介绍：

```text
Amazon/ERP 导出文件
→ Python 字段清洗、类型转换和多表关联
→ GMV/ACOS/CVR 等确定性计算
→ 按站点/类目/ASIN/日期聚合
→ 日报与趋势图表
→ 飞书多维表格/机器人
→ DeepSeek 归纳解释
→ 人工确认
```

#### 报表处理工具

重点说明工具解决了什么，而不是只说“用了 pandas/Python”：

- 原始文件字段、日期格式和数值类型不统一，需要先标准化。
- 销售、库存和广告数据粒度不同，需要明确关联键与统计时间窗口。
- GMV、ACOS 和 CVR 由程序确定性计算，不能交给大模型算。
- 生成结果按业务维度聚合，让运营人员能定位具体站点、类目和 ASIN。
- 对异常值、缺字段和分母为零保留检查或人工复核，避免生成“看起来正常”的错误日报。

可能追问及回答：

**工具的输入输出是什么？**  
输入是销售、库存和广告导出文件，以及 ERP 中需要关联的业务字段；输出是统一指标口径的结构化日报和趋势图表，并同步到飞书协作入口。不要声称输入来自实时 SP-API，除非当时确实如此。

**如何做多表关联？**  
先明确每张表的粒度，再选择稳定的 SKU/ASIN、站点和日期窗口作为关联维度。广告数据还要注意 campaign、搜索词和归因时间，不能把不同粒度直接 join 后相加。

**如何保证计算正确？**  
对关键指标保留公式和口径；用小规模样本与原 Excel 结果核对；检查总额、行数、日期范围和分母为零；随机抽取 ASIN 回溯到原始记录。若当时没有自动化测试，应坦诚说明以人工抽样和汇总核对为主，并把这件事作为个人项目加入数据契约和测试的原因。

#### 飞书自动化

可以解释为三个层次：

1. 多维表格承担结构化结果和状态记录。
2. 审批流承担人工确认和责任边界。
3. 机器人承担通知与入口，不在机器人里隐藏核心指标计算。

如果被问到 ERP 对接，说明数据通过 OpenAPI 在业务系统与飞书协作入口之间传递；重点讲字段映射、失败提示、重复通知和权限，不要泛化为“实现了完整 ERP 平台”。

#### AI 日报为什么只做草稿

DeepSeek 接收的是已经由代码计算好的指标和必要上下文，负责归纳趋势、组织语言和给出待审核解释。它不负责重新计算 GMV/ACOS，也不自动发布业务结论。这样可以降低幻觉、错误归因和未经确认信息进入团队群的风险。

#### 工作经历与个人项目如何衔接

推荐回答：

> 工作中我先用 Python、飞书和模型 API 解决报表与日报效率问题，但也发现导出文件口径漂移、重复同步、API 限流、模型输出缺字段和审核留痕等问题。个人项目不是简单重写工作脚本，而是把这些痛点进一步抽象为数据契约、幂等事务、Multi-LLM Gateway、LangGraph 状态机和 Human-in-the-Loop，并用离线测试覆盖失败路径。

### 3.1 销售与流量数据链路

讲解顺序：

```text
创建报表 → 轮询状态 → 下载/GZIP 解压 → Pydantic 校验
→ raw 入库 → 标准化 upsert → 更新游标 → 指标/告警 → 人工处理
```

关键设计：

- Token 刷新用并发保护，避免多个协程同时刷新并造成惊群。
- 限流按 operation 管理，因为 Amazon usage plan 不是所有接口共享同一个固定 QPS。
- 429 优先尊重 `Retry-After`，可恢复 5xx 使用带全抖动的指数退避。
- raw、标准化数据和同步游标放在同一事务；中途失败整体回滚。
- 稳定业务键负责幂等，raw 文件哈希、request ID 和 trace ID 支持回放与审计。

常见追问：

**为什么不是 Exactly once？**  
外部 HTTP、任务队列和数据库无法只靠一句配置提供端到端 Exactly once。我采用至少一次交付，加数据库唯一键/upsert、事务和可回放原始层，让重复执行得到相同业务结果。

**为什么用全抖动？**  
如果多个 worker 同时指数退避，它们仍可能在同一时刻重试。全抖动把等待时间随机分散，降低重试风暴。

**如何防止报表口径错误？**  
区分 raw、标准层和指标层；用 Pydantic 检查字段、日期、币种和数值；对导入结果做 reconciliation，并保留输入文件哈希和时间窗口。

### 3.2 Listing 决策链路

```text
可信竞品数据 → 保留 source_id/observed_at → 并发生成三版
→ 每版严格五条德语 bullet → 确定性合规检查 → 人工 approve/reject/edit
```

为什么使用 LangGraph：

- 价值在于显式状态、节点级可观测、checkpoint 和可中断恢复。
- 生成与规则检查职责分开，便于独立测试。
- 人工节点是状态机的一部分，而不是在 Prompt 末尾写一句“请人工检查”。

为什么硬规则不用 LLM：

- 标题长度、五点数量、禁词和绝对化表述是可确定判断的问题。
- LLM 适合生成和解释，不适合为自己的输出做唯一裁判。
- 法律适用性、类目例外和证据不足的情况继续交给人工。

失败路径：

- Provider 超时或 5xx：由 Gateway 尝试下一个 Provider。
- JSON 缺字段或 bullets 数量不对：Pydantic 验证失败并进入 fallback。
- 事实没有来源：合规结果标记阻断或要求人工补充证据。
- 所有模型失败：返回安全错误，不发布、不静默吞错。

### 3.3 广告解释链路

```text
Ads Reporting v3 → ACOS/CTR/CVR/CPC/TACOS
→ 检查样本量和归因窗口 → 查询库存/价格/自然销售
→ 生成假设与证据评分 → 飞书待审建议
```

指标定义：

- `ACOS = 广告花费 / 广告销售额`
- `ROAS = 广告销售额 / 广告花费`
- `CTR = 点击量 / 展示量`
- `CVR = 订单量 / 点击量`
- `CPC = 广告花费 / 点击量`
- `TACOS = 广告花费 / 总销售额`

回答广告问题时必须带时间窗口、样本量和归因延迟。ACOS 上升不等于应立即降价或停词，可能来自 CPC 上升、CVR 下降、库存/价格变化、促销结束或广告订单尚未归因。

## 4. Multi-LLM Gateway 高频题

### 4.1 为什么要有 Gateway

业务 Agent 不应直接依赖不同厂商的请求格式和错误语义。Gateway 统一 API、模型别名、结构化输出、超时、并发、熔断、fallback、指标与安全错误，Provider 特有逻辑留在 Adapter 内。

### 4.2 Structured Output 如何保证

三层保证：

1. 只允许请求服务端注册的 Schema，避免客户端提交任意复杂 Schema。
2. 将 JSON Schema 发给支持该能力的 Provider。
3. 收到响应后仍使用 Pydantic 二次解析；非法 JSON、缺字段或约束失败都视为 Provider 失败。

不要回答“Prompt 里要求返回 JSON”就算结构化输出。

### 4.3 哪些错误应该 fallback

- 超时、网络错误、可恢复 5xx、非法结构化输出：可以 fallback。
- 认证失败、账户配额或权限错误：应告警并停止盲目降级，避免掩盖配置问题。
- 所有 Provider 失败：返回不含 Key、堆栈和 Prompt 全文的安全 503。

### 4.4 熔断与并发闸门区别

- Semaphore 限制同时在途请求数，避免资源被高延迟模型调用耗尽。
- Circuit breaker 根据连续失败暂时跳过故障 Provider，减少无意义请求并加快失败响应。
- Timeout 限制单次调用占用资源的最长时间。

三者解决的问题不同，通常一起使用。

### 4.5 模型路由怎么做

业务使用稳定别名，例如 `listing-quality`，具体模型 ID 由环境变量配置。路由考虑任务质量、Schema 能力、成本、延迟和可用性，不在业务代码里散落易过时的模型名。

## 5. RAG、Prompt 与 MCP 高频题

### 5.1 RAG 为什么需要版本和生效时间

Amazon 规则和内部 SOP 会变化。文档应包含 `document_id`、版本、生效/失效时间、marketplace、category、language、access_scope 和来源。检索时先做权限与有效期过滤，防止过期规则或越权文档进入上下文。

### 5.2 如何评估 RAG

先评估检索，再评估答案：

- Recall@5：正确证据是否进入前五。
- MRR：第一个正确结果排得是否靠前。
- 引用正确率：回答是否真正被引用内容支持。
- 过期规则泄漏率与权限过滤。
- 拒答正确率：证据不足时能否输出 `insufficient_evidence`。

### 5.3 为什么实时订单不放向量库

订单状态和销量是结构化、实时、需要精确权限控制的数据，应该通过只读业务工具或 SQL 查询。RAG 更适合规则、SOP 和说明文档。

### 5.4 Prompt 如何资产化

Prompt 记录 ID、版本、marketplace、category、输入/输出 Schema。固定评测集覆盖西服和宠物地毯两个类目，CI 对 Schema 通过率和硬规则设门槛，主观质量只报告趋势。

### 5.5 MCP 是不是安全边界

不是。MCP 只是工具交互协议。真正的安全来自最小工具面、认证上下文、scope、参数校验和审计。项目只暴露销售指标、库存风险、政策检索和 Listing 草稿四个窄工具，seller/marketplace 由可信上下文注入，也没有发布、改价或广告写工具。

## 6. 飞书与 Human-in-the-Loop

飞书不是展示层，而是运营人员接管 Agent 的控制面：

- Bitable 以稳定 `source_key` search + update/create，重复任务不产生重复记录。
- 事件入口验证 verification token，并记录操作者和 trace ID。
- 卡片展示指标口径、时间范围、来源和限制。
- 高风险动作只生成建议或草稿，由人 approve/reject/edit。
- 只有状态或严重度变化才再次通知，降低告警疲劳。

如果被问“为什么不全自动”，回答：Listing 发布、改价和广告预算会直接产生业务与合规风险。自动化收益不足以覆盖错误执行的损失，因此先实现建议、证据、审批、审计和失败补偿，再讨论有限范围自动化。

## 7. C++ 与 Python 技术迁移

### 7.1 Reactor 与 asyncio 的共同点

- 都围绕事件驱动和非阻塞 I/O。
- 都要避免在事件循环中执行长时间阻塞任务。
- 都需要背压、资源上限、超时和优雅关闭。

### 7.2 主要差异

- C++ epoll/Reactor 更接近系统调用、连接生命周期和内存管理，控制力强但实现成本高。
- Python asyncio 用协程表达异步流程，适合大量 I/O 等待，但 CPU 密集任务受 GIL 和事件循环阻塞影响。
- LLM 服务的瓶颈通常是外部网络等待和上游限流，因此 async 很合适；CPU 密集解析应拆到线程池、进程或独立 worker。

### 7.3 如何回答“数千并发”

只有在能说明测试环境、连接模型、消息频率、P95 和资源使用时才说“压测支撑数千并发”。如果没有完整压测证据，应说“面向数千连接设计”，并重点解释 Reactor、线程模型、背压和连接管理。

## 8. 系统设计：从 1 个 Seller 扩展到 100 个 Sellers

### 8.1 单 Seller

- 一个 Gateway、一个 worker、PostgreSQL、Redis。
- 定时任务按报表窗口触发。
- seller 信息来自服务端配置，不由模型传入。
- 指标、原始文件哈希、游标和审核记录统一保存。

### 8.2 100 个 Sellers

- 所有任务携带可信 tenant context，数据库表按 seller 维度建立唯一键和必要索引。
- 限流器按 seller/application + operation 隔离，避免一个 Seller 拖慢全局。
- 调度加入公平队列、优先级、并发配额和随机抖动。
- worker 水平扩展；幂等键和事务保证重投安全。
- 大型原始报表转对象存储，数据库保留元数据、哈希和处理状态。
- Gateway 根据 Provider/tenant 设置并发与预算，记录 token、cost、P95 和 fallback_count。
- 告警按 tenant 聚合，关键操作保留审计日志和人工审批。

### 8.3 容量估算回答框架

先问四个参数：Seller 数、每个 Seller 的报表频率、单文件行数/大小、模型请求量。再估算峰值任务数、API 限流、数据库写入量、模型并发与成本。不要脱离工作负载直接报 QPS。

## 9. 业务场景题

### 9.1 GMV 上升是否代表运营变好

不一定。需要同时看毛利、退款率、广告花费、库存周转、自然单占比和促销影响。GMV 是规模指标，不是利润指标。

### 9.2 ACOS 下降是否一定是好事

不一定。可能因为减少了探索性投放，导致总销售额或新品曝光下降。结合 TACOS、自然销售、利润率、预算利用率和生命周期判断。

### 9.3 CVR 下降如何排查

按漏斗拆解：流量来源是否变化、搜索词意图是否偏移、价格/优惠是否变化、库存与配送时效、Listing 内容和评价、变体关系、竞品动作。先验证数据时间范围与样本量，再给归因。

### 9.4 如何优化德国站 Listing

先收集可信的产品事实、目标关键词和竞品证据；区分标题、五点和 A+ 的信息职责；使用自然德语而非关键词堆砌；检查绝对化宣传、医疗声明和 GPSR 相关信息；最终由熟悉类目和德语的人员审核。

### 9.5 AI 建议与 GMV 增长如何证明因果

不能只用上线前后对比。应设计可比时间窗口或 A/B 测试，控制价格、促销、库存和季节性，并预先定义 GMV、CVR、利润和退款率指标。当前项目没有这类真实线上实验，所以不声称平台直接带来 GMV 增长。

## 10. 行为面试素材

用 STAR 结构准备，每个故事控制在 2 分钟。

### 10.1 主动发现并解决重复报表问题

- **S**：运营日报来自多个导出文件，字段和口径不同。

- **追问准备**：具体减少了哪些步骤？如何验证数字没有算错？没有量化数据时不要编造节省比例。

### 10.2 从运营问题抽象为平台能力

- **S**：单脚本只能处理一次任务，难以应对重试、重复执行和模型失败。

## 11. 高压追问与参考回答

**这是不是一个 Demo？**  
当前是可离线验证的核心骨架，不是已生产上线系统。它超出一次性 Demo 的地方在于有明确模块边界、失败处理、幂等、事务、权限、人工审核和回归测试；仍缺真实账号和长期运行证据，我会明确区分。

**为什么测试都是 mock？**  
CI 不能依赖真实 Amazon、飞书或模型账号，否则不确定、成本不可控且有密钥风险。mock 外部协议边界验证业务逻辑；真实账号仍需要独立 smoke test 和验收记录，两者不能互相替代。

**为什么不用一个大 Prompt？**  
大 Prompt 难以定位失败，也无法稳定保证事实来源、三版本输出、规则检查和人工中断。拆节点后可以对每一步设置 Schema、重试、测试和审计。

**为什么不直接使用 LangChain 的所有封装？**  
只在图状态和检索抽象有明确收益时使用框架。限流、事务、业务指标和权限保持为普通可测试代码，避免核心规则依赖隐式框架行为。

**如何防 Prompt Injection？**  
模型不能决定 tenant、seller 或 scope；工具输入经过 Pydantic；敏感上下文由服务端注入；只暴露窄的只读/草稿工具；无发布和广告写工具；所有调用留 trace 和审计。

**如果 Provider 都挂了怎么办？**  
返回安全 503，保留 request ID，指标记录失败与 fallback 次数；任务可由上层按幂等键重试或转人工，不用未经验证的自由文本作为降级结果。
